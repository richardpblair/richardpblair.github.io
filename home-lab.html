Breaking Things Safely: A Virtual Cybersecurity Home Lab

I built a virtual cybersecurity home lab using Oracle VirtualBox and a pfSense firewall so I can practice networking, system administration, traffic analysis, and security testing — without my neighbors calling the FBI because my “learning” looks suspicious from the outside.

pfSense Windows 11 Ubuntu Server Kali Linux Windows Server 2022

Objective

The goal was to simulate a small enterprise network behind a firewall, using multiple operating systems and roles: workstation, internal server, attacker box, and an Active Directory environment.

  • Windows 11 client workstation
  • Ubuntu Server (internal services + SSH)
  • pfSense firewall/gateway controlling traffic
  • Kali Linux attacker VM for future testing
  • Windows Server 2022 for Active Directory (next phase)
ref

Host Specs

  • CPU: AMD Ryzen 9 9950X
  • RAM: 64 GB DDR5-6000
  • Storage: 4 TB NVMe (RAID-5)
  • VirtualBox: 7.2.4

Translation: my PC is basically a small datacenter with feelings.

Lab Topology

pfSense sits between the “Internet side” (NAT/WAN) and the internal lab network (LAN). Everything in the lab lives on the internal network named Cyber.

Device OS IP Address Role
pfSense FreeBSD 10.0.0.1 Firewall / Gateway
Windows 11 Windows 11 10.0.0.10 Client / Workstation
Ubuntu Ubuntu 24.04 LTS 10.0.0.20 Internal Server
Kali Kali Linux 2025.3 10.0.0.30 Attacker VM
Windows Server Windows Server 2022 10.0.0.40 Active Directory (Next Phase)

Build Process (Step-by-Step)

I mapped out the network first: pfSense on the edge, internal systems behind it on the Cyber network. This made it easy to keep the lab isolated and consistent.

Deliverable: a network diagram + IP plan before creating any VMs.

I installed VirtualBox and created each VM with reasonable resources and the right network mode. Every internal VM used Internal Network: Cyber. pfSense used two adapters: one for WAN/NAT and one for LAN/Cyber.

Windows 11

  • 4096 MB RAM, 2 CPUs
  • 80 GB disk
  • Network: Internal Network (Cyber)

Ubuntu 24.04 LTS

  • 2048 MB RAM, 2 CPUs
  • 25 GB disk
  • Network: Internal Network (Cyber)

Kali Linux

  • 4096 MB RAM, 2 CPUs
  • 25 GB disk
  • Network: Internal Network (Cyber)

Windows Server 2022

  • 4096 MB RAM, 2 CPUs
  • 50 GB disk
  • Network: Internal Network (Cyber)

pfSense (Netgate)

  • 2048 MB RAM
  • 20 GB disk
  • Two adapters: WAN (NAT) + LAN (Internal Network: Cyber)

OS installs were straightforward. After install, I updated systems and verified pfSense had WAN and LAN configured. I also installed VirtualBox Guest Additions to make resizing work smoothly (ultrawide monitor problems).

Tip: In VirtualBox: Devices > Insert Guest Additions CD image… then follow prompts and reboot.

I verified IP addresses and connectivity between systems, then confirmed traffic was flowing through pfSense.

Commands Used

  • Windows: ipconfig
  • Ubuntu: ip addr show
  • Ping test: ping 10.0.0.20

Result

Successful pings between Windows and Ubuntu confirmed the internal network was working and routed correctly via pfSense.

Once networking was stable, I tested remote access and outbound routing through pfSense.

Windows → Ubuntu (SSH)

Command:

ssh boston@10.0.0.20

Ubuntu → Internet (Ping)

Command:

ping 8.8.8.8

What I Learned

  • VirtualBox networking modes are the make-or-break of any lab setup.
  • pfSense is both awesome and intimidating (like a final boss with a firewall UI).
  • Documentation + screenshots make rebuilding labs way easier.

Next Steps (Planned)

  • Promote Windows Server to a Domain Controller and join Windows 11 to the domain.
  • Practice DHCP, DNS, and Group Policy in the lab.
  • Simulate attacks with Kali Linux inside a safe subnet.
Safety Note: All testing stays in an isolated subnet behind pfSense. No “oops” packets go wandering into the real world.
